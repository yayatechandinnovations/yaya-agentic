# Milestone M2.8 — Tenant Registry & Cross-Tenant Profile Clone

| | |
|---|---|
| **Goal** | Promote the tenant from an implicit row to a first-class configuration aggregate: explicit registration, lifecycle (active / suspended / archived), a single host base URL that governs both outbound HTTP tool dispatch and inbound request origins, and a `cloneProfile(src → dst)` operation that lifts a profile plus its transitive dependencies into another tenant. |
| **Timeline** | ~3–4 days backend + ~2 days Flutter (slotted between M2.5 and M3 in practice; conceptually post-M2.7 since it supersedes that doc's leniency) |
| **Status** | 🟩 Done |
| **Depends on** | M1 (admin CRUD, profiles/capabilities/tools schema), M2 (engine + SSE), M1.5 (operator-auth gate on `/v1/admin/**`) |
| **Design refs** | [`docs/design/tenant-registry-design.md`](../design/tenant-registry-design.md) — entire doc. ## Scope (IN)

- Tenant becomes a configuration aggregate: `host_base_url`, `host_base_url_allowlist`, `inbound_origin_allowlist`, `require_https`, `default_authenticator_binding_id`, `default_recording_strategy_id`, lifecycle status, audit timestamps. (§3)
- Explicit registration via `POST /v1/admin/tenants`. The implicit `ensureTenant` auto-create on first reference is removed; admin writes against unknown tenants fail with `unknown_tenant`. (§4)
- Lifecycle endpoints: suspend / resume / archive. Archived tenants reject writes and tool dispatch; the row stays for audit and the id can never be reused. (§4.1)
- Tenant health endpoint surfaces misconfiguration (missing host base, dangling binding/recording-strategy refs) per-row. (§4.1)
- HTTP tool `urlTemplate` is **path-only**. Save-time validation rejects absolute / protocol-relative / non-`/`-prefixed inputs with `absolute_url_not_permitted`. (§6.1)
- Multi-environment as a first-class supported flow via `host_base_url_allowlist` + `X-Yaya-Host-Base-Url`. One tenant row owns N approved bases; the header chooses among them, never introduces a new trust root. (§6.3.1)
- Inbound origin allowlist enforcement on `/v1/sessions/*`. Empty list = host base URL origin only; non-empty = glob match. Denials audited. (§5)
- One-shot `AbsoluteToPathMigrator` for legacy HTTP tools whose absolute URL origin matches the tenant's host base — operator-confirmed dry-run before apply; mismatched origins surface as `ORIGIN_NOT_TENANT_HOST` and are never silently rewritten. (§9.2)
- `CloneService` walks profile → capabilities → tools → knowledge bindings → auth binding → recording strategy → personality, builds a plan, applies atomically. Conflict / knowledge-location / personality policies configurable per request. (§7)
- `tenant_clone_jobs` audit row for every clone (dry-run or applied), capturing the resolved plan. (§7.7)

## Out of scope (NOT here)

- Knowledge corpus migration. `CloneService` copies the source *definition* only; the destination re-ingests under its own egress / API keys. Preserves embedding provenance. (§7.5, §11.1)
- Cross-tenant *linking*. Two tenants never share a row in `capabilities` / `tools` / `knowledge_sources`. Every cross-tenant transfer goes through `CloneService`. (§11.3)
- Tenant deletion of historical recorded sessions — that's RTBF and ships with M5.
- `audit_admin_writes.clone_job_id` per-write back-references — `audit_admin_writes` doesn't exist yet; `tenant_clone_jobs.plan_json` captures the full plan, which is sufficient until the audit surface lands. (§7.7 note)
- A separate operator-role check on the clone endpoint — the existing M1.5 operator-auth gate on `/v1/admin/**` is what applies. A dedicated per-tenant operator-role model lands with M5.
- In-flight session drain semantics on `SUSPENDED` beyond "no new sessions accepted." A `DRAINING` intermediate status slots in later if needed; no schema change required. (§11.5)

## Backend deliverables

### B8.1 — V9 migration: tenant columns + constraints

Adds `display_name` (renamed from `name`), `status`, `host_base_url`, `host_base_url_allowlist`, `inbound_origin_allowlist`, `require_https`, `default_authenticator_binding_id`, `default_recording_strategy_id`, `updated_at`, `created_by`, `archived_at`. Composite FK `(id, default_authenticator_binding_id) → auth_bindings(tenant_id, id)` is `DEFERRABLE INITIALLY DEFERRED` to break the chicken-and-egg between tenant + binding creation. Also adds `audit_inbound_origin_denied`. Existing rows backfill to `status=ACTIVE` with `host_base_url=NULL` — flagged on the health endpoint, never silently defaulted.

### B8.2 — `TenantEntity` + repository expansion

Typed `Status` enum (ACTIVE / SUSPENDED / ARCHIVED). Repository adds `findActiveById`, `findByIdAndStatus`, `findAllByStatus`.

### B8.3 — Tenant CRUD + lifecycle endpoints + health

`POST/GET/PUT/PATCH /v1/admin/tenants[/{id}]`, `POST .../suspend|resume|archive`, `GET .../health`. Validation pipeline: slug shape, URL grammar, `require_https` flag, referenced binding/recording-strategy exists. Stable error codes (`bad_tenant_id`, `bad_host_base_url`, `unknown_auth_binding`, `tenant_id_taken`, `tenant_archived`, …). Health endpoint reports per-dependency state + warnings.

### B8.4 — Path-only validation on HTTP tool writes

`BaseUrlValidator.validatePathOnlyTemplate` is the canonical enforcement point. `AdminController.validateHttpSpec` calls it on every write; the dispatch-time `BaseUrlResolver` keeps its same input contract but loses its "URL is already absolute → pass through" branch (now unreachable for new writes). Legacy rows surface on the tenant health endpoint and refuse to dispatch until rewritten.

### B8.5 — `AbsoluteToPathMigrator` + endpoint

`POST /v1/admin/tools/migrate-to-path?tenant=&dryRun=`. Walks the latest version per HTTP tool, classifies as `candidate` (origin matches tenant `host_base_url`) or `unsafe` (`ORIGIN_NOT_TENANT_HOST` / `TENANT_HAS_NO_HOST_BASE_URL`). `dryRun=true` returns the plan; `dryRun=false` writes new versions for every candidate. Unsafe entries are never touched.

### B8.6 — Inbound origin allowlist filter

`OriginEnforcer` on session endpoints. Missing `Origin` header → skip (server-to-server / non-browser pass-through). Tenant unknown → skip (auth layer rejects downstream). Empty allowlist → permit `host_base_url`'s origin only. Non-empty allowlist → glob match. Denial → 403 `origin_not_permitted` + audit row.

### B8.7 — `CloneService` (dry-run + apply)

Walks the full dependency closure from a source profile. Conflict policy: `FAIL` / `SKIP` / `NEW_VERSION` per resource. Knowledge location strategy: `RETAIN` (warn) / `TEMPLATE` (substitute source tenant id → destination) / `OMIT` (null out). Personality policy: `AUTO` (fill gaps) / `ALWAYS` / `NEVER` (require destination has one). Apply runs inside a single transaction; partial failures roll back the entire clone. Knowledge sources land `UNINDEXED` at the destination — re-ingest required.

Endpoint: `POST /v1/admin/tenants/{src}/profiles/{profileId}/{version}/clone`.

### B8.8 — V10 migration: `tenant_clone_jobs`

UUID PK, source/destination tenant + profile id + version, `plan_json`, `status` (DRY_RUN / APPLIED / FAILED), `error_json`, `applied_at`, `applied_by`. Indexed by destination, source, and `(status, created_at)` for retention queries.

### B8.9 — Cross-tenant denial + path-only integration tests

`CloneServiceTest` covers: dry-run plan shape, apply with path-only preservation, `FAIL` conflict (409), `NEW_VERSION` bump, same-tenant rejection, unknown-profile 404, personality `NEVER` rejects missing destination, personality `AUTO` creates per-locale when missing. `AbsoluteToPathMigratorTest` covers candidate vs unsafe classification and the apply path. `TenantControllerTest` covers registration / validation / lifecycle / health / unknown_tenant guard / path-only at the tool-write boundary / origin allowlist allow + deny.

## Flutter deliverables (`yayaagenticweb`)

### F8.1 — Tenants list + detail + create wizard

`/admin/tenants` becomes the platform's home screen (slot #2 in the nav rail). List shows status pill, host base URL, and a live health badge per tenant. Detail tile expands to a two-step create-or-edit form: identity (slug, display name) → routing (host base URL, allowlists, requireHttps). Live API-driven validation; field errors bind to the backend's stable codes.

### F8.2 — Tenant health card

Embedded in the expanded tenant tile. Reads `/v1/admin/tenants/{id}/health` (a Riverpod family provider keyed by id, invalidated on every tenant write and migrator run). One green/red row per check (`hostBaseUrlSet`, `authBindingResolves`, `recordingStrategyResolves`) plus a warnings list and a "Migrate absolute URLs → path" action that opens a confirm-before-apply dry-run diff dialog.

### F8.3 — Profile clone wizard

`Clone to another tenant…` action on every profile tile. Three-step modal: configure (destination tenant + conflict / knowledge / personality policies) → review plan (per-resource action chips with `fromVersion → toVersion`, per-action notes, warnings card) → apply (success state with the audit job id and a one-line reminder that knowledge sources land UNINDEXED at the destination).

### F8.4 — Tools form path-only helper + extract-path affordance

`_PathOnlyUrlField` replaces the bare URL `TextField` on the tools form. Inline error text + warning row when the value is absolute or protocol-relative. `Extract path` button appears only when the URL's origin matches the tool tenant's `host_base_url` — anything else has to be resolved manually, mirroring the migrator's safety guarantee.

## Acceptance criteria

- [x] Creating a tenant with a missing or malformed `host_base_url` is rejected at save time with `bad_host_base_url`.
- [x] Saving an HTTP tool with an absolute `urlTemplate` is rejected at save time with `absolute_url_not_permitted`.
- [x] Existing tools with absolute URLs surface on the tenant health endpoint and refuse to dispatch with a clear error.
- [x] `AbsoluteToPathMigrator` rewrites only tools whose absolute origin matches the tenant host; the rest carry an `ORIGIN_NOT_TENANT_HOST` note and are left alone.
- [x] An inbound `/v1/sessions/*` request whose `Origin` doesn't match `inbound_origin_allowlist` (or the host base URL origin when the list is empty) returns 403 `origin_not_permitted` with an audit row.
- [x] Cloning `support-agent @ v3` from `acme` to a fresh `acme-eu` produces working profile + capabilities + tools at version 1 with no manual URL editing; the tool's `urlTemplate` survives verbatim and the destination tenant's `host_base_url` resolves at dispatch time.
- [x] A clone with `conflictPolicy=FAIL` against a destination that already has one of the resources returns 409 `destination_resource_exists` and **no** rows are written.
- [x] A clone with `knowledgeLocationStrategy=TEMPLATE` substitutes the source tenant id in `S3Prefix.uri` (and other location fields); the source row is preserved untouched.
- [x] `tenant_clone_jobs` carries the full plan and an `applied_by` value for every applied clone; dry-runs are recorded with `status=DRY_RUN`.
- [x] Two clones of the same source profile into different destination tenants (`acme-eu`, `acme-apac`) resolve to their respective `host_base_url`s at dispatch — no cross-talk, no copied secrets.
- [x] Admin writes against an unknown tenant fail with 400 `unknown_tenant` rather than silently auto-creating.

## Risks & open questions

- **Path-only tightening breaks demos / fixtures.** Mitigation: only the M0 retail demo's fixtures live in this repo, and they use BEAN handlers — nothing absolute to rewrite. External operator fixtures get the `AbsoluteToPathMigrator` with a tenant-host check.
- **Operators surprised by explicit registration.** Mitigation: the tenant health surface flags missing config; the M2.8 admin landing screen lists tenants by status so misconfigured ones are visible at a glance.
- **CloneService becomes a hot spot for "small additions."** Mitigation: every new resource type lands as an explicit flag on `CloneRequest` — never silent. Personality, for instance, gates behind `personalityPolicy` and defaults to AUTO so existing voices are never overwritten.
- **Inbound origin enforcement breaks legitimate non-localhost integrations.** Mitigation: the allowlist supports globs (`https://*.acme.com`); operators populate it before flipping enforcement on. Initial release ships the filter "enforce on present `Origin`" — server-to-server callers that omit `Origin` are unaffected.
- **Auth secrets at the destination.** Cloned `auth_bindings` reference an authenticator + chain, never copy secrets. The destination must provision its own `tenant_secrets` row before dispatch will succeed.
